The **implementation** (in the working tree diff) correctly handles this -- the new `calculateVisibleRange` comment on line 229 says "yOffsets 是遞減排列" and the `calculateCurrentIndex` binary search already handles descending order. ### 2. Fix 2 has a critical timing gap: `reflectScrolledClipView` fires BEFORE `isZooming` is set

The implementation of Fix 2 only sets `csView.isZooming = true` inside `scrollViewMagnificationDidChange` (line 1776 of ImageViewController). However, the call chain for pinch/scroll zoom is:

1. `magnify(with:)` / `handleCmdScrollZoom` -> `setMagnificationPreservingInsets` -> `setMagnification` 
2. AppKit synchronously calls `reflectScrolledClipView` -> `updateVisibleSlots` (**`isZooming` is still `false` here!**)
3. Control returns to `magnify(with:)` -> `scrollDelegate?.scrollViewMagnificationDidChange` -> sets `csView.isZooming = true` -> calls `updateVisibleSlots` again

This means the **first** `updateVisibleSlots` call per magnification event runs with `isZooming == false`, allowing slot recycling to happen and causing the very flicker the fix is supposed to prevent.

### 3. Keyboard zoom (`zoomIn`/`zoomOut`) does not set `csView.isZooming` at all

The `zoomIn()` and `zoomOut()` methods at `/Users/pc035860/code/cee/Cee/Controllers/ImageViewController.swift:972-997` call `setMagnificationCentered`, which calls `scrollView.setMagnification`. This triggers `reflectScrolledClipView` -> `updateVisibleSlots` with `csView.isZooming == false`. Neither `setMagnificationCentered` nor `zoomIn`/`zoomOut` ever sets `csView.isZooming = true`. The magnification delegate is not called in this path. Keyboard zoom will still exhibit the black flicker.

## Plan patch (proposed edits)

### Replace Fix 2 with a more robust approach

Instead of setting `isZooming` only in `scrollViewMagnificationDidChange` (which fires too late), the flag should be set **before** `setMagnification` is called. Two options:

**Option A -- Set `isZooming` in `ImageScrollView` before `setMagnification`:**

In `ImageScrollView.magnify(with:)` and `handleCmdScrollZoom`, set `continuousView.isZooming = true` **before** calling `setMagnificationPreservingInsets`. This way, when `reflectScrolledClipView` fires synchronously, the flag is already set. The delegate callback then handles the end-of-gesture cleanup.

Similarly, in `ImageViewController.setMagnificationCentered`, wrap the `scrollView.setMagnification` call with `csView.isZooming = true` ... `csView.isZooming = false` + `updateVisibleSlots` cleanup.

**Option B (simpler, recommended) -- Set `isZooming` at the `reflectScrolledClipView` level:**

In `ImageScrollView.reflectScrolledClipView`, check if `magnification != 1.0` or if a magnification gesture is in progress and set the flag there. However, this couples the scroll view to the content view's state too much.

**Option C (simplest, recommended) -- Move the `isZooming` guard into `ImageScrollView.reflectScrolledClipView`:**

Since `ImageScrollView` already calls `continuousView.updateVisibleSlots`, it can check `ImageViewController.isZooming` (via the existing delegate or a property) and set `csView.isZooming` before the call. But this requires either exposing `isZooming` from the controller or passing it through the delegate.

**Concrete recommendation:** Add the `isZooming` set/clear to `setMagnificationCentered` in `ImageViewController`:

```swift
private func setMagnificationCentered(_ targetMagnification: CGFloat) {
    isZooming = true
    continuousScrollContentView?.isZooming = true  // ADD
    defer {
        isZooming = false
        // Don't clear csView.isZooming here for gesture zoom (handled by delegate);
        // DO clear for keyboard zoom (no gesture phase)
    }
    ...
}
```

And in `zoomIn` / `zoomOut`, after `setMagnificationCentered` returns, do:

```swift
if settings.continuousScrollEnabled {
    continuousScrollContentView?.isZooming = false
    continuousScrollContentView?.updateVisibleSlots(for: scrollView.contentView.bounds)
}
```

### Replace Fix 3 description

Fix 3 should **not** remove the `updateVisibleSlots` call from either location. Both are needed:

## Risks & mitigations

- **Risk**: `activeSlots` grows unbounded during long pinch-zoom gestures because recycling is suppressed. If user zooms out significantly (making many images fit on screen) and then pans while still in the gesture, many slots could accumulate.
  **Mitigation**: The post-zoom cleanup call (`isZooming = false` + `updateVisibleSlots`) handles this. But consider adding a hard cap (e.g., `activeSlots.count > visibleRange.count * 3`) as a safety valve even during zoom.

- **Risk**: If `scrollViewMagnificationDidChange` is never called with `.ended`/`.cancelled` (e.g., AppKit event sequence anomaly), `isZooming` could remain `true` indefinitely, preventing all recycling.
  **Mitigation**: Add a timeout-based fallback. Or, more simply, clear `isZooming` on the next `updateVisibleSlots` call that comes from `reflectScrolledClipView` with no active magnification gesture (check `scrollView.magnification` hasn't changed since last call).

- **Risk**: `calculateCurrentIndex` returns `max(0, min(count-1, high))` when `scrollY` falls outside all image bounds. During zoom, `visibleRect.maxY` could exceed `totalHeight`, returning index 0, while `visibleRect.minY` could be below 0, returning `count-1`. This would produce `firstVisible=0, lastVisible=count-1` -- the entire image set, defeating the purpose of buffer limits.
  **Mitigation**: This is actually correct behavior -- if the viewport genuinely spans the entire document, all images should be visible. But verify the binary search edge cases with unit tests for extreme zoom-out scenarios.

- **Risk**: The `isSlotActive(for:)` check is O(n) linear scan on `activeSlots` array. With many accumulated slots during zoom, this could degrade the inner loop of `manageSlotViews`.
  **Mitigation**: Consider using a `Set<Int>` (`activeSlotIndices`) alongside the array for O(1) lookup. Low priority since `activeSlots` is typically small (< 20).
